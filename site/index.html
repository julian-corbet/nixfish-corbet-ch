nixfish

Your fish config, as code — safely adopted, not overwritten.

programs.fish.enable = true already exists in home-manager. What's missing is the part nobody writes down: on a real machine, config.fish is almost never empty. nixfish is the documented, verified adoption sequence — a one-time backup flag, never force = true — plus the one piece of fish state that genuinely can't be declared the normal way.

Status: pre-alpha, one real module, one real migration

✓

Extracted from an actual migration, not a demo

One real module (fish), documenting a real migration — a vendor fish config and a personal dotfile pile, both plain hand-maintained files, moved to Nix for the first time. The core claim (a pre-existing config.fish makes home-manager switch refuse rather than silently overwrite, and -b <ext> is the correct fix, not force = true) was verified by reading home-manager's actual module source, not assumed from docs.

!

Universal variables can't be a managed file

set -U state lives in a separate persisted file home-manager's own fish module never touches. universalVariables converges it via an idempotent shellInit command instead — correct, but it means nixfish never "owns" that file the way it owns config.fish, and can't detect drift in it the way a managed file would.

!

No test suite yet

Not exercised against every home-manager version in the wild — verified against the one pinned revision this was actually built and migrated on.

What it does

Safe adoption, not silent overwrite

Flip programs.fish.enable blind on a machine with a real config.fish and home-manager switch refuses to activate — correct, but a wall, not a plan. nixfish documents the actual sequence: activate once with a backup flag, diff the backup against what you expected, confirm a fresh shell still works, then delete the backup. Never a standing default.

Declarative universal variables

universalVariables takes a plain attrset (prompt colors, pager colors, anything normally set via set -U) and converges it as idempotent shellInit commands — the one corner of fish state that has no managed-file equivalent in home-manager's own module.

Composes with your own config

programs.fish.shellInit, .shellAliases, and .functions stay entirely yours — nixfish's own contributions merge alongside them (types.lines merges across multiple contributions) rather than replacing anything.

Zero coupling to Arch, Wayland, or forwarding

Deliberately independent of nixarch and nixremote — the adoption problem this solves has nothing to do with either. Only depends on nixpkgs, so it can reach anything with a shell.

Get started

1. Add the flake input

inputs.nixfish.url = "github:julian-corbet/nixfish-corbet-ch";

2. Import and declare

imports = [ inputs.nixfish.homeManagerModules.fish ];

nixfish.fish = {
  enable = true;
  universalVariables = {
    fish_color_normal = "F0F0F0";
    fish_color_command = "22C55E";
  };
};

3. First-time adoption (existing config.fish)

home-manager switch --flake .#you@host -b hm-bak

Diff the .hm-bak file against what you ported, confirm a fresh shell works, then delete the backup. Never leave backupFileExtension configured as a standing default.

Roadmap (planned, not yet built)

A home-manager-free sibling

For machines where home-manager itself would be disproportionate weight (a minimal survival-mode VPS). Deliberately not built yet — the one fleet case that prompted the question was better served by a couple of lines directly in that project's own modules, with no cross-repo dependency. Worth revisiting if a second genuine case shows up.

Declarative bash adoption

programs.bash has the identical wholesale-ownership model and the identical -b/force mechanics — the same documentation applies, just not yet written up as its own option surface here.

Integration tests

Exercising the actual collision/backup behavior against a real pre-existing file, not just eval-time type checks.
